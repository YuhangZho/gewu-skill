# 冷启动渠道矩阵

## 一句话定位
> 它属于开源项目冷启动的渠道层，是用来解决"项目做好了发到哪里、怎么发才有人用"的——不是找地方发帖，是有策略的渠道组合。

## 🎯 核心收获 · 重点知识
**一句话成果**：学完你现在能给 gewu-skill 设计一个"测试场→迭代场→扩量场"三渠道错峰矩阵，每个渠道收不同维度反馈驱动不同对象迭代，制造 star 增量尖峰+持续长尾。

**重点结论**（AI 与你达成的最终共识）：
- **渠道矩阵 ≠ 找渠道宣发**——"找渠道宣发"是单向信息输出；渠道矩阵是有策略的渠道组合，三层价值：①覆盖面（不同渠道触达不同人群）②迭代测试（A/B 测试哪个渠道转化高）③流量节奏（错峰发制造持续讨论长尾，不是一天爆发后死掉）。核心是组合策略，不是渠道清单。
- **发链接 vs 建立信任**——发链接=把渠道当广告牌；建立信任=把渠道当社区。Reddit/V2EX/掘金都要求先贡献再索取，新号直接发广告=spam=被 ban=渠道永久关闭。冷启动本质是口碑传播，口碑要求信任，信任要求先贡献再索取。不是"包装"项目，是真诚分享你解决的问题和过程。
- **每个渠道反馈不同→触发不同迭代对象**——矩阵真正价值不是"多渠道覆盖"，是"多渠道多维度反馈驱动多轮迭代"。Linux.do 技术深度用户反馈→改 Features/机制说明；掘金开发者反馈→改 Tech Stack/技术细节；少数派效率工具用户反馈→改 Demo/场景故事。如果 3 渠道内容一样，反馈也会一样，迭代只能改一个对象=浪费矩阵价值。
- **迭代具体迭 3 样东西**（不是笼统"改商品"）：①README 首屏标语/定位语（反馈"看不懂这是啥"）②Quick Start/安装步骤（反馈"装不上/跑不起来"）③Demo/Features 展示（反馈"看懂了但不想用"）。渠道矩阵的反馈=告诉你 README 7 节点漏斗哪个节点在漏水。
- **测试场→迭代场→扩量场三渠道错峰**——第一个渠道是测试场（不是首发场），linux.do 实战论断："先在一个渠道打磨好话术和 README 转化，再复制到下一个"。每个渠道间隔 2-3 天，制造持续讨论长尾。渠道B反馈回到迭代动作形成第二个迭代循环，不是只有渠道A触发迭代。
- **Reddit 风控红线**——反营销、反自我推销文化；养号→参与讨论→再发；选对子版块；保持 IP 稳定。V2EX 创意板块可发但标题不能太广告化；Hacker News Show HN 必须"Show HN: [名] - [描述]"格式且评论区亲自回复；掘金会限流纯推广最好技术分享包装。
- **Product Hunt 时机**——榜单机制=一天一轮太平洋时间 0 点重置，工作日日榜第一需 600-800 票。太早上（没准备好）=转化低排名掉机会浪费；太晚上（已发过其他渠道）=用户"见过了"不投票。最佳=定位+README+Demo 都优化好后集中一天发多渠道冲日榜前五拿官方徽章。一年最好只上一次。
- **gewu-skill 渠道矩阵初稿**：Linux.do（测试场·先发，分享设计哲学+费曼五步工程化）→掘金（迭代场·第二，技术实现+Agent Skills 标准）→少数派（扩量场·第三，场景故事+我学习时遇到的问题）。3 渠道间隔 2-3 天错峰。

**重点知识大纲 / 脑图**：
- 渠道矩阵本质
  - 不是找渠道宣发，是组合策略
  - 三层价值：覆盖面 / 迭代测试 / 流量节奏
  - 核心=组合策略不是渠道清单
- 发链接 vs 建立信任
  - 广告牌逻辑 vs 社区逻辑
  - 先贡献再索取（养号/参与讨论/真诚分享）
  - 新号直接发广告=spam=被ban
- 三渠道错峰策略
  - 渠道A·测试场（先发，听反馈）
  - 反馈1→迭代→README转化提升
  - 渠道B·迭代场（第二，复制优化版）
  - 反馈2→反馈环回到迭代（第二个迭代循环）
  - 渠道C·扩量场（第三，场景故事扩量）
  - star增量尖峰+持续长尾
- 迭代具体3样东西
  - README首屏标语（"看不懂"）
  - Quick Start安装（"装不上"）
  - Demo/Features（"不想用"）
- 每渠道反馈不同触发不同迭代
  - Linux.do→改Features/机制
  - 掘金→改Tech Stack/技术细节
  - 少数派→改Demo/场景故事
- 3个失效边界
  - 同时发=跳过迭代=浪费渠道
  - 直接广告=被ban=渠道关闭
  - 受众不匹配=来了不留
- 主要渠道禁忌
  - Reddit：养号/反营销/IP稳定/选对子版块
  - V2EX：标题不能太广告
  - Hacker News：Show HN格式+亲自回复
  - 掘金：技术分享包装，纯推广限流
  - Product Hunt：一年一次，最佳=优化好后冲日榜

## 📝 轻笔记
<details>
<summary>📝 轻笔记（点开看推演摘要与卡壳细节）</summary>

**起点认知**：找渠道宣发。
→ 校准：渠道矩阵 ≠ 找渠道宣发，是组合策略（覆盖面+迭代测试+流量节奏三层）。

**视觉模型（五问 · 用户先讲）**：
- **对象**：渠道本身、每个渠道反馈、迭代动作、README 转化率、star 增量
- **连接**：渠道→反馈→迭代→转化率→star
- **流向**：用户初答"同时发"→校准为错峰发+反馈环
- **边界**：直接广告
- **变化**：好会涨星，不好就迭代

**视觉模型结构（AI 校准后）**：
- **chart_type（AI 自主判定）**：flow（测试场→迭代场→扩量场+反馈环+3失效边界）
- **节点 kind**：process（渠道A/B/C、迭代动作）、data（反馈1/2）、rule（README转化率）、state（star尖峰）、feedback（持续长尾）、boundary（3个失效边界）
- **结构纠偏**：用户初版流向答"同时发"→校准为错峰发+反馈环；用户验收时跳过渠道B反馈回到迭代的环→补丁"每个渠道都是一次迭代机会"。

**类比 1（连锁店）**：先开一家而不是同时铺开，从第一家接受反馈改店里商品，再开第二家继续收反馈，最后开第三家扩量。不是只推广而是真正帮助每家店铺营利积累口碑，一家家开是为了不断验证和优化。

**类比 2（发微信不同群）**：先发一个群看反馈优化，再发一个群，而不是上来一起发。

**大白话讲解（讲给外行）**：它属于开源运营里的问题，是用来解决"项目做好了发到哪里、怎么发才有人用"的。好比开连锁店——不是同时铺开所有店，而是先开一家，听顾客反馈改店里的招牌/点单流程/橱窗陈列，改好了再开第二家继续听反馈再改，最后开第三家扩量。每家店收到的反馈不一样（第一家顾客关心菜单、第二家关心价格、第三家关心装修），所以改的东西也不一样。关键是不能同时开——同时开就没法根据反馈改了。也不是去每家店门口发传单就走，而是真正把每家店经营好，让顾客愿意口口相传。

**三重验证**：
1. **正向验证（Vue.js）**：Evan You 先 Google 内部测（改API）→Hacker News Show HN（改定位）→Twitter Build in Public（改传播话术）→中英文双语文档（改上手门槛）。每步反馈改不同对象，不是同时铺所有渠道。
2. **失效边界补丁**：直接广告容易写成软广而不是真诚分享 → 补救：以"我做了个东西想听反馈"心态发，不是"推广产品"心态发。
3. **换类比（发微信不同群）**：先发一个群看反馈优化，再发一个群，而不是上来一起发。

**卡壳点复盘**：
- 卡壳 1：以为渠道矩阵=找渠道宣发 → 补：是组合策略，三层价值（覆盖面/迭代测试/流量节奏）。
- 卡壳 2：发链接 vs 建立信任只说"无法聚焦" → 补：发链接=广告牌逻辑，建立信任=社区逻辑，先贡献再索取。
- 卡壳 3（最深）：3 渠道内容方向一样 → 补：每渠道受众不同→反馈维度不同→触发不同迭代对象，矩阵价值=多维度反馈驱动多轮迭代。
- 卡壳 4：迭代太笼统"改商品" → 补：具体3样——README标语/Quick Start/Demo Features。
- 卡壳 5：验收跳过反馈环 → 补：渠道B反馈回到迭代形成第二个循环，每个渠道都是迭代机会。

**成长对比**：
- 起点：找渠道宣发
- 现在：渠道矩阵是组合策略（覆盖面+迭代测试+流量节奏）；发链接 vs 建立信任（先贡献再索取）；每渠道反馈不同触发不同迭代；测试场→迭代场→扩量场三渠道错峰；gewu-skill 初稿=Linux.do→掘金→少数派。

### 1. Evan You · 「开源渠道矩阵实战代表」
- **身份**：Vue.js 创始人、VoidZero 创始人
- **与本概念的关系**：Vue.js 是渠道矩阵的经典案例——Evan You 没有同时铺所有渠道，而是 Google 内部测→HN→Twitter→文档长尾，每步反馈改不同对象。
- **核心洞见**：Vue 成功不只是技术好，更是"文档优先 + 多语言 + 响应速度 <12h + 社区自治"的组合策略——每个渠道的反馈都转化为对应对象的迭代（内部改API、HN改定位、Twitter改传播、文档改上手门槛）。
- **代表作**：Vue.js（2014-）
- **扩展阅读**：[尤雨溪谈Vue.js](https://zhuanlan.zhihu.com/p/58335278) · [Evan You 官网](https://evanyou.me/)

### 2. 孟健 · 「海外冷启动渠道清单整理者」
- **身份**：AI编程领域博主、掘金作者
- **与本概念的关系**：整理了海外产品免费宣传渠道清单（V2EX/HN/Reddit/PH/社交媒体/Startup平台），实测有效，适合冷启动阶段。
- **核心洞见**：冷启动本质不是到处发链接，而是找到早期用户建立信任——邮件标题要吸引但不标题党，正文简洁说清解决什么问题，CTA明确，及时回复反馈。
- **代表作**：《产品做完了不会推广？这份海外冷启动渠道清单》
- **扩展阅读**：[掘金原文](https://juejin.cn/post/7560958498676031534)

### 3. Pieter Levels · 「Build in Public 渠道方法论代表」
- **身份**：独立开发者、Nomad List/Maker Log 创始人、《MAKE》作者
- **与本概念的关系**：他是 Build in Public 文化的代表人物——不靠渠道矩阵铺开，而是靠 Twitter 持续公开分享开发过程，把社交媒体变成"测试场+迭代场+扩量场"三合一。
- **核心洞见**：Build in Public 本身就是一种渠道策略——每天分享进度=持续在社区贡献价值=积累信任=发产品时自然有人关注。不是"发链接"，是"让人看着你做出来"。
- **代表作**：《MAKE: The Indie Maker Handbook》
- **扩展阅读**：[Pieter Levels Twitter](https://twitter.com/levelsio) · [Maker Log](https://makerlog.io/)
