# 项目定位与价值主张

## 一句话定位
> 它属于开源项目冷启动的地基层，是用来解决"为什么有些项目发出去没人理"的。

## 🎯 核心收获 · 重点知识
**一句话成果**：学完你现在能给 gewu-skill 写一句让陌生访客 3 秒内识别"跟我有关"的定位语，并理解为什么定位决定传播链能否启动。

**重点结论**（AI 与你达成的最终共识）：
- **定位不是两个独立东西，是三件套**：为谁 + 痛点 + 独特价值，缺一件就立不住。缺"独特价值"= 凭什么用你而不用替代品没说清。
- **痛点普遍性是传播链的开关**：star 和转发是社交行为，访客大脑第一反应是"跟我有关吗"。相关性可识别 → 转发链延续；痛点不普遍 → 传播链在第一个节点就断。
- **star 留人 vs 转发拉新是两件事**：star = 访客变"己方"标记，未来 release 可唤醒；转发 = 把项目推给下一批同质人群，才是传播链燃料。star 涨但转发为 0 = 飞轮没燃料 = 冷启动失败。
- **GitHub 是被动展示不是主动算法推荐**：Trending 主信号=过去 24h star 增量（非总 star），日粒度 UTC 00:00 重置，≤7 天新仓库 ×1.6 加权。GitHub 冷启动比 B 站难得多——B 站转发→算法助攻，GitHub 转发→无算法助攻→必须人肉分发到外部渠道。
- **Atlassian 电梯演讲模板**：For [谁] who [想要什么], [项目] is a [类别] that provides [核心收益]. Unlike [替代方案], it [差异化点]。30 秒-1 分钟，禁术语，必含受众/需求/项目/价值四件。
- **gewu-skill 定位校准**：原版"把有问题问 AI 变成一张有罗盘的学习地图"——好的是对比结构和"罗盘"意象，缺的是"为谁"和"凭什么用你"。参考版："把'有问题问 AI'，变成一张你给目标、它给依赖顺序、带你费曼到底的学习地图"。
- **5 个失效边界**：太宽（适用人群什么都覆盖）、太窄（只服务自己）、自说自话（术语堆砌）、与替代品无差异（又一个 X）、不可验证（说最好用但没法立刻试）。

**重点知识大纲 / 脑图**：
- 三件套结构
  - 为谁（目标用户，要具体不能"所有"）
  - 痛点（要普遍才可识别）
  - 独特价值（凭什么用你不用替代品）
- 传播机制（为什么定位决定冷启动）
  - 定位语 → 访客 3 秒相关性识别
  - 识别成功 → 价值判断 → star 留人 + 转发拉新
  - 转发 → 新访客触达（反馈环）→ 传播链启动
  - 两个失效边界：痛点不普遍→链断第1节点；无差异→只 star 不转发→链不启动
- GitHub 发现机制（被动 vs 主动）
  - Trending = 过去 24h star 增量，日粒度重置
  - ≤7 天新仓库 ×1.6 加权
  - 无算法推荐 → 必须人肉分发到外部渠道
- Atlassian 电梯演讲模板
  - For who + want + is a + provides + Unlike + it
  - 30 秒-1 分钟，禁术语，四件必含
- 5 个失效边界与补丁
  - 太宽 → 具体到人群
  - 太窄 → 找共性痛点
  - 自说自话 → 大白话替换术语
  - 无差异 → 点出凭什么
  - 不可验证 → 给 demo/截图

## 📝 轻笔记
<details>
<summary>📝 轻笔记（点开看推演摘要与卡壳细节）</summary>

**起点认知**：定位是用户特质与用户的规模，价值主张是解决的痛点。定位是提供什么服务针对什么人群。项目介绍就是介绍特色。
→ 校准：定位 ≠ 价值主张是两个独立东西，是三件套（为谁+痛点+独特价值）。

**视觉模型（五问 · 用户先讲）**：
- **对象**：访客大脑（问能帮我什么忙）、项目页面（前三段）、star/观望
- **连接**：访客眼睛 ↔ README；访客大脑 ↔ 内容价值判断
- **流向**：README 首段 → 向下读完 → 决定试用 → 试用安装 → 反馈 star
- **边界**：需要安装 python 的非技术人员
- **变化**：是否需要 python 决定是否试用

**视觉模型结构（AI 校准后）**：
- **chart_type（AI 自主判定）**：flow（访客决策流程 + 反馈环 + 失效分支）
- **节点 kind**：concept（定位语）、actor（访客大脑）、rule（相关性识别、价值判断）、process（star+转发）、feedback（新访客触达）、state（传播链启动）、boundary（2 个失效边界）、concept（ChatGPT 参照系）
- **结构纠偏**：用户初版把"是否需要 python"当主要变化点——这是技术门槛边界，不是定位机制核心。AI 校准为"定位语→相关性识别→价值判断→star+转发→反馈环"的主流程，技术门槛归入失效边界。

**类比 1（B 站）**：github 项目冷启动好比刷 B 站——标题封面=定位语，1 分钟抓住眼球才点赞=star，特别共鸣才转发=传播燃料。关键差异：B 站转发→算法助攻，GitHub 转发→无算法→必须人肉分发。

**类比 2（发朋友圈）**：图=定位语（第一眼决定是否看下去），文字=价值传递（有深度有共鸣才点赞），点赞=star，转发=传播链燃料。

**大白话讲解（讲给外行）**：它属于开源运营里的问题，是用来解决"为什么有些项目发出去没人理"的。好比刷 B 站看到不认识的 up 主——第一眼扫描标题封面（=定位语），感兴趣才点进去，1 分钟抓住眼球才点赞（=star），特别共鸣才转发，只有转发才触发更多人看。但 GitHub 不像 B 站会自动推荐，只有被动推荐机制（Trending 榜按 24h star 增量排），所以也得找准为谁解决什么问题、为什么是 gewu 而不是其他工具。

**三重验证**：
1. **正向验证（VS Code）**：为程序员解决编程环境分散的问题，集成 IDE 编程到编译一站式落地。三件套齐全。
2. **失效边界补丁**：定位太宽（"为所有学习者"）→ 改成"想系统学新领域但没框架的自学者"→ README 首屏只写这一句，不写"所有""各种""每个人"。
3. **换类比（发朋友圈）**：图=定位语、文字=价值传递、点赞=star、转发=转发。

**卡壳点复盘**：
- 卡壳 1：以为定位=用户特质+规模、价值主张=痛点是两个独立东西 → 补：是三件套，缺独特价值。
- 卡壳 2：以为"开源生态"是痛点普遍性决定性因素的原因 → 补：真正机制是相关性可识别→传播链可启动，开源是分发模式不是传播开关。
- 卡壳 3：把 star 和转发混成"新 star 决定推流" → 补：star=留人，转发=拉新，两件事不能混。GitHub 不因 star 多推流。
- 卡壳 4（最深）：以为 GitHub 完全不推 → 补：GitHub 是被动展示（Trending 按 24h star 增量），不是完全不推。B 站主动算法推荐 vs GitHub 被动展示，冷启动难度差异巨大。

**成长对比**：
- 起点：定位=用户特质+规模，价值主张=痛点（两个独立东西，缺独特价值）
- 现在：定位是三件套（为谁+痛点+独特价值），决定传播链能否启动；star 留人 vs 转发拉新不能混；GitHub 被动展示非主动推荐，必须人肉分发点燃传播链。

## 🔭 视野拓展 · 行业洞见

<summary>🔭 视野拓展 · 行业洞见（点开看多维度权威参照）</summary>

### 1. Geoffrey Moore · 「定位框架理论之父」
- **身份**：硅谷顾问、《跨越鸿沟》作者
- **与本概念的关系**：他提出的"电梯演讲模板"是今天 Atlassian 模板的理论源头——For [who] who [need], [product] is a [category] that [benefit]. Unlike [competitor], it [differentiation].
- **核心洞见**：定位的本质不是描述产品，而是**在用户心智中占据一个位置**——你必须先回答"相对于替代品，你占据的是哪个位置"，否则就是"又一个 X"。
- **代表作**：《跨越鸿沟》（Crossing the Chasm, 1991）
- **扩展阅读**：[Crossing the Chasm 摘要](https://www.blinkist.com/books/crossing-the-chasm-en) · [Geoffrey Moore 官网](https://geoffmoore.com/)

### 2. Evan You · 「开源项目定位实战代表」
- **身份**：Vue.js 创始人
- **与本概念的关系**：Vue 的定位是开源项目定位的经典案例——"渐进式框架"三个字同时回答了"为谁"（想要渐进采用的开发者）、"痛点"（React/Angular 太重）、"独特价值"（可逐步引入，不必全盘重构）。
- **核心洞见**：Vue 成功的关键不是技术最强，而是**定位最清晰**——Evan You 多次在访谈中说"我不跟 React 比，我只服务好'想要渐进式'的那群人"。文档优先 + 多语言 + 响应速度 <12h，都服务于这个定位。
- **代表作**：Vue.js（2014-）
- **扩展阅读**：[Evan You: The State of Vue](https://www.youtube.com/watch?v=4nWi8xR8NmQ) · [Vue.js 官网](https://vuejs.org/)

### 3. Atlassian Team · 「电梯演讲工程化方法论」
- **身份**：Atlassian 团队协作工具公司，Team Playbook 作者
- **与本概念的关系**：把 Moore 的理论框架工程化成可执行的 6 步 Play——从"团队对齐要 pitch 什么"到"压力测试"到"最终版落地"，每步有具体动作和检查清单。
- **核心洞见**：好定位不是一个人憋出来的，是团队**压力测试**出来的——读回初稿后检查"受众清晰吗/问题清晰吗/价值相关吗/可信吗/外部人能懂吗/够简洁吗/有强记忆点吗/下一步清晰吗"8 个 checkpoint。
- **代表作**：Atlassian Team Playbook（开源的团队协作方法论库）
- **扩展阅读**：[Elevator Pitch Play](https://www.atlassian.com/team-playbook/plays/elevator-pitch) · [Atlassian State of Teams 2025](https://atlassianblog.wpengine.com/wp-content/uploads/2025/03/the-state-of-teams-2025.pdf)
