# Skill

## 一句话定位
> 它属于 AI 工程，是一套**按需读取的说明书 / SOP**：把人的专家经验固化成规范，指导 Agent 用统一、可预期的方式处理某类任务。

## 动态图像（视觉建模）
用户提问 → AI 拿这个任务，跟库里每个 skill 的 description「读一遍」做**语义匹配**（不是字符串关键词引擎，是模型自己判断"这件事像不像该用它"）→ 命中则把该 skill 的说明书**读进当前上下文** → 按说明书的流程与标准执行、产出 → 用完放回，并**不会永久改变模型能力**。
关键画面：skill 平时躺在库里，**用到那一刻才被拿出来当指令照着做**。

## 类比
像「专业手册 / SOP」。AI 是干活的人，skill 是他临时翻开的那本手册——手册不会自己干活，是人拿着手册干。
反例（澄清常见误解）：它**不**像游戏 DLC（装上就永久长在身上），那是**微调模型**才有的"天生变强"。

## 大白话讲解
skill 不是给 AI"永久升级能力"，而是给它一份"这类活该怎么干"的说明书，用到时现读现用。
靠每个 skill 自带的一段 description 触发——description 把"我是干嘛的、什么时候该用我"写清楚，模型据此语义判断要不要调它。所以 description 写得好不好，直接决定 skill 会不会在该出场时被想起来。

## 三个应用例子
1. 嵌入式工程师：把"嵌入式代码 + 汇报"的固定规范做成 skill，AI 每次按同一套结构输出。
2. 亚马逊运营：把 listing 优化的最佳实践固化成 skill，产出稳定贴合平台与预期。
3. 文档处理：skill 里写"这一步去跑这段转码/抽表脚本"，把机械步骤交给脚本，judgment 留给 AI。

## 失效边界（什么时候不该用 skill）
- 需要**真正执行外部动作**（发邮件、查数据库、接云端服务）→ 用 [[MCP]] / 工具，不是 skill。
- 某步是**死协议、固定输入输出、无需动脑** → 那是**脚本**；skill 可以"雇佣"脚本，但 skill ≠ 脚本。
- 想让 AI **天生就擅长**某领域、不靠现读说明书 → 用**微调模型**。
- skill 的甜区：用户已有经验，用一层"约束 + 指导"规范 [[Agent]] 的处理方式，避免发散、固化输出范围与表达，从而得到更符合预期的结果。

## 卡壳点复盘
- 一开始以为触发是"关键词字符串匹配" → 其实是模型拿任务跟 description 做**语义匹配**。
- 以为 skill 像 DLC"永久变强" → 其实是**按需读取的指令**，用完不改变模型本身。
- "skill 里也能放脚本，那俩差在哪？" → 差在**谁做判断**：脚本是固定协议、精确产出；skill 是可商量的建议约束，中间过程靠 AI judgment。skill 可调用脚本完成机械子步骤。
